三、为什么总是这些 bug — 四个结构性根因

壹

单口径数据模型 對抗 多源现实

ChildItem 的注释自己写着「根据物料类型，只有一个有值」——但 CLAUDE.md 的反模式表早就写明：同一物料可以同时在 PRD_INSTOCK + 采购入库 + 委外入库有收货，必须 union，永不单源。后端其实把 5 个来源都算出来了（_make_row 全填进 BOMJoinedRow），却在 _bom_row_to_child 这一刀按「物料类型」只保留一个口径、其余丢弃。于是每一个路由错误（类别焊死/is_purchase/前缀）都自动升级成数据丢失。P0·1、P0·3、今天的 bug，全是这一个形状。

贰

前端把「数据被丢了」伪装成「该列不适用」

列按 type-code 显示 '-'。用户分不清「这列对该物料不适用」和「数据其实有、但被路由丢了」——所以这些 bug 全靠业务同事人肉对金蝶才暴露（这次也是）。系统自己永远不报警。

叁

同一套语义写了两遍，每个 wave 只修一格

aux 三层回退、收货去重、类别路由、需求口径——live Python 和 cache SQL 各实现一遍。审计抓到三对「半边修复」：收货去重 cache 有 live 没有（P0·2）、03.% SUM cache 有 live 没有（P1·8）、is_purchase cache 有 live 没有（今天）。而且每次都是声称等价的注释 + 不执行真路径的守卫陪着它们上线。

肆

默认值兼职信号 + 在 cache 上验证、在 live 上运行

False/0/'' 既是「没填」又是「真值」（is_purchase=False、Decimal(0)、category=''），任何一条路径漏填就静默走错分支。而修复的端到端验证做在 cache 数据上，prod 默认跑 live——验证的不是用户跑的。

这不是 13 个 bug，是一个架构债的 13 次利息：数据模型只允许一个口径，现实却有五个来源——路由层每错一次，真数据就消失一次，而界面和测试都配合着保持沉默。

四、建议杀法（按序）

顺序动作了结
①现在扩展今天的修复：is_purchase 接通（原计划）＋ 自制分支同样按 is_purchase 分流（主料+外购 → 采购口径）——同一把刀，多劈一个分支今天的 bug + P0·1
②本周预警链路：销售出库/订单 UNIQUE 加 entry 粒度（照抄领料表 9c5fe58 的修法）+ 补 per-MTO DELETE；live 收货侧 Tier-2 接上已有的去重状态P0·4 + P0·2
