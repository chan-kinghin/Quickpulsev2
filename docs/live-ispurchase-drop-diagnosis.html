二、断链在哪 — live 数据流

① Kingdee · 数据都在

PPBOM 读数已映射 FMaterialId.FIsPurchase（factory.py:321）
合成行已有批量主数据查询 lookup_material_categories（client.py:460）— 只查了类别，没查 FIsPurchase

↓这些料不在 PPBOM → 走合成行 2b（采购单）/ 2d（领料）
② 断点 · _make_row（mto_handler.py:892-934）

构造 BOMJoinedRow 时没有传 is_purchase

模型默认 is_purchase: bool = False（cache_reader.py:75）→ live 路径上每一行都是 False，不管金蝶主数据怎么说。Step 1 BOM 行、2a/2b/2c/2d 合成行全部如此。

↓带着假的 is_purchase=False 进入路由
③ 误触发 · _bom_row_to_child:1682（上周 b0e9647 新增分支）

类别=外销包材 + is_purchase=False → 自制包材口径

该分支是给吸塑/跟型件（真自制包材）修的：material_type=SELF_MADE → 前端只显示生产入库列（=0）+ 领料列；采购订单数量、采购入库数量整列丢弃。真外购的镜盒彩贴们全军覆没。

↓两个可见症状
症状 A · 数据归零（03.23.x / 03.45.x）

包材自制 0 订单数量 · 0 入库数量 · 只有领料 · 0% —— 截图 1 的 385-389 行，本地 handler 基线已逐行复现。

症状 B · chip 错标（01.11 莱卡布等）

副 chip 按 item.is_purchase 渲染（dashboard.html:472）→ live 上永远显示「自制」。莱卡布因类别覆盖被过滤（05bd364 只允许非自制类别覆盖）幸运地走了采购口径，数字对、标签错。

为什么上周没发现 · 信任债窗口 b0e9647（6-04）是在 cache 数据上端到端验证的 —— cache 行有 synced 的 is_purchase，所以验证通过。但默认路径自 Phase 3（5-30 上线）起是 live。live builder 是 Pre-Change Checklist「3 条数据路径」之外的第 4 条路径。症状 A 在 prod 存活 6 天（6-04 → 6-10）；症状 B（chip 全错）自 live 切换起 ~11 天。bug-patterns：Pattern 1 第 6 次 + Pattern 8 振荡下半周期（修自制包材 → 打破外购包材）。

三、修复 — 把已有的值接通（不动路由逻辑、不动前端）

Fix 1 · client.py

lookup_material_categories 增查 FIsPurchase → 返回 {code: (类别, is_purchase)}。多 org 重复行：任一真即真。

↓
Fix 2 · _fetch_live

拆两个 map：category_by_code（保持现有过滤）+ is_purchase_by_code（全量不过滤）。

↓
Fix 3 · _make_row 增加 is_purchase 参数

Step 1 BOM 行 ← PPBOM 链 first.is_purchase
2a/2c/2d ← is_purchase_by_code.get(code, False)
2b ← is_purchase_by_code.get(code, True)（采购单来源，定义即外购）

守住上周的修复 吸塑/跟型件（真自制包材，IsPurchase=False）拿到的还是 False → 仍走生产入库口径，b0e9647 不回退。回归测试两边都加。
