① 你看到的现象

销售订单空 ＝ 正常（包材不是成品，本就不会出现在销售订单）。真正的问题是中间那栏：采购订单 = 0，于是没有任何「需求基线」，完成率永远算成 0%。

② 数据真相：需求其实存在，只是被丢掉了

查本地缓存（同款物料、其它 MTO 实测）——它们在 BOM 里有真实 need_qty，根本不是「没需求」：

○ 关键事实
缓存里共 541 条「外销包材 + 自制(is_purchase=0)」物料，其中 540 条有 need_qty。这是一个系统性的口径缺口，不是个例。

③ 根因：包材一栏只认「采购订单」，而自制包材没有采购单

物料主数据 · BD_MATERIAL
CategoryID.FName = 外销包材
同时 IsPurchase = False（吸塑/跟型件是厂内自制，不是外购）
↓

→ effective_type = 2，挂上「包材」chip（chip 是对的：包材是类别，不是采购标志）
↓
ChildItem 组装 · effective_type==2 分支
需求列 ← purchase_order_qty（采购订单数量）
这条分支只读采购侧字段，不读 need_qty / 生产入库。对外购包材没问题，对自制包材是致命的——
↓
结果：采购订单 = 0
自制件没有采购订单 → 需求基线为 0
need_qty = 170,880 被丢弃
真实需求挂在 row 上，分支没读 → 永远不显示
完成率 = 0 / 0 → 0%

根因位置：src/query/mto_handler.py:1682 — elif effective_type == 2: 分支

④ 为什么「外购包材」没事，「自制包材」就坏了

外购包材 · is_purchase = True
外箱 / 内盒 / 纸卡
有采购订单 → purchase_order_qty 有值 → 需求、累计入库、完成率全部正确。当前逻辑天然适配。✓
自制包材 · is_purchase = False
吸塑 / 跟型内衬 / 对折盒
无采购订单。需求应来自 need_qty（BOM 用量 × 母件数量），齐套应看生产入库而非采购入库。当前逻辑读错了源头。✗

⑥ 修复（已实现）

前端早有「双键」设计：chip + 筛选认 material_type 名称，列 + 语义认 material_type_code 整数。所以一处后端分流即可，前端零改：

包材子类code (口径)name (chip)需求列完成率
外购 (is_purchase=1)2包材采购订单数量（不变）入库/订单
自制 (is_purchase=0)1包材生产入库.应收 = need_qty实收/need_qty
改动点：src/query/mto_handler.py — elif effective_type == 2 and type_label == "包材" and not row_is_purchase（仅当确认是 外销包材 类别时分流，fallback 路径不动）
✓ 真实数据验证（AS2603021-5 · 03.02.07.006 EP-5耳塞热压吸塑）
经真实缓存 JOIN 跑通：TYPE = 1 / 包材，需求(应收) = 21,600（修复前是空/0），采购订单 = 0。951 单测全绿。
